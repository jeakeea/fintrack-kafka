## Описание

| Параметр | Значение |
|----------|----------|
| **URL\*** | `/transfers/{payment_id}` |
| **Метод HTTP\*** | `GET` |
| **Наименование метода\*** | Получение статуса перевода (состояния саги) |
| **Частичный запрос атрибутов\*** | Нет |
| **Постраничный вывод\*** | Нет |
| **Возврат информации о количестве записей (count)\*** | Нет |
| **Управление сортировками\*** | Нет |
| **Сортировка по умолчанию\*** | Не применяется (возвращается один объект) |
| **Способ авторизации\*** | Без авторизации (аутентификация клиента вне scope учебного стенда, см. FR-W1) |
| **Роли\*** | Для всех ролей |
| **Ссылка на swagger** | http://localhost:8000/docs (живой Swagger UI gateway); контракт – [`openapi.yaml`](../openapi.yaml) |
| **http-статус коды в ответе\*** | 200 – ок; Ошибки: 404, 500 |

## Выходные параметры\*

| Параметр | Тип | Описание | Обязательный | Пример |
|----------|-----|----------|--------------|--------|
| payment_id | string (uuid) | Идентификатор платежа | Да | `3f8c1a9e-…` |
| status | string | Статус саги: `INITIATED` \| `ANTIFRAUD_PENDING` \| `LEDGER_PENDING` \| `COMPLETED` \| `FAILED` | Да | `COMPLETED` |
| step | string | Последний зафиксированный шаг саги: `INITIATED` \| `ANTIFRAUD_REQUESTED` \| `LEDGER_REQUESTED` \| `COMPLETED` \| `FAILED` | Нет | `COMPLETED` |
| from_account | string | Счёт-источник | Нет | `acc_ivan` |
| to_account | string | Счёт-получатель | Нет | `acc_maria` |
| amount | integer (int64) | Сумма перевода в минорных единицах (копейки) | Нет | `150000` |
| reason | string | Причина отказа для `status=FAILED` (`amount_limit`, `velocity`, `insufficient_funds`), иначе `null` | Нет | `null` |
| updated_at | string (date-time) | Момент последнего перехода саги | Да | `2026-07-05T12:00:02+00:00` |

## Логика работы

1. Прочитать строку таблицы `saga_state` по первичному ключу `payment_id`.
2. Если строка не найдена – ответ `404 {"detail": "payment not found"}`.
3. Если найдена – вернуть поля записи как есть (см. «Выходные параметры»).

Метод только читает состояние; статус меняет оркестратор по мере переходов
саги. Система согласована в конечном счёте (eventual consistency): сразу после
`202 Accepted` метод может вернуть промежуточный статус (`INITIATED`,
`ANTIFRAUD_PENDING`, `LEDGER_PENDING`); терминальные значения – `COMPLETED`
либо `FAILED` с заполненным `reason`.

## Коды ответа\*

| HTTP-статус | Код сообщения | Сообщение | Сценарий возникновения |
|-------------|---------------|-----------|------------------------|
| 200 | OK | Операция выполнена успешно | Перевод найден, состояние возвращено |
| 404 | NotFound | `payment not found` | Перевод с указанным `payment_id` не существует |
| 500 | InternalServerError | Сервис временно недоступен | Внутренняя ошибка сервера (например, недоступна БД) |
