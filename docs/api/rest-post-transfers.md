## Описание

| Параметр | Значение |
|----------|----------|
| **URL\*** | `/transfers` |
| **Метод HTTP\*** | `POST` |
| **Наименование метода\*** | Создание P2P-перевода (запуск саги) |
| **Способ авторизации\*** | Без авторизации (аутентификация клиента вне scope учебного стенда, см. FR-W1) |
| **Роли\*** | Для всех ролей |
| **Ссылка на swagger** | http://localhost:8000/docs (живой Swagger UI gateway); контракт – [`openapi.yaml`](../openapi.yaml) |
| **http-статус коды в ответе** | 202 – заявка принята; Ошибки: 400, 422 |

## Входные параметры\*

| Параметр | Тип | Описание | Расположение | Обязательный | Значение по умолчанию | Пример |
|----------|-----|----------|--------------|--------------|----------------------|--------|
| Idempotency-Key | string | Уникальный ключ операции, генерируется клиентом (рекомендуется UUID v4). Гарантирует ровно одно создание перевода при ретраях | Header | Да | | `9f1c2a3e-7b54-4c1d-9a2b-0f3e4d5c6a7b` |
| from_account | string | Счёт-источник (списание). Используется как ключ партиционирования событий саги | Body | Да | | `acc_ivan` |
| to_account | string | Счёт-получатель (зачисление) | Body | Да | | `acc_maria` |
| amount | integer (int64) | Сумма перевода в минорных единицах валюты (копейки), строго больше нуля. 1500,00 ₽ = `150000` | Body | Да | | `150000` |
| currency | string | Валюта перевода (ISO-4217) | Body | Нет | `RUB` | `RUB` |

## Пример запроса

```http
POST /transfers HTTP/1.1
Host: localhost:8000
Content-Type: application/json
Idempotency-Key: 9f1c2a3e-7b54-4c1d-9a2b-0f3e4d5c6a7b

{
  "from_account": "acc_ivan",
  "to_account": "acc_maria",
  "amount": 150000,
  "currency": "RUB"
}
```

## Выходные параметры\*

| Параметр | Тип | Описание | Обязательный | Пример |
|----------|-----|----------|--------------|--------|
| payment_id | string (uuid) | Идентификатор созданного платежа; для повторного запроса – идентификатор ранее созданного | Да | `3f8c1a9e-2b54-4c1d-9a2b-0f3e4d5c6a7b` |
| status | string | `accepted` – создан новый перевод, сага запущена; `duplicate` – ключ уже использован, возвращён существующий `payment_id` | Да | `accepted` |

## Пример ответа

```json
{
  "payment_id": "3f8c1a9e-2b54-4c1d-9a2b-0f3e4d5c6a7b",
  "status": "accepted"
}
```

Повторный запрос с тем же `Idempotency-Key`:

```json
{
  "payment_id": "3f8c1a9e-2b54-4c1d-9a2b-0f3e4d5c6a7b",
  "status": "duplicate"
}
```

## Логика работы

1. Проверить наличие заголовка `Idempotency-Key`; если он не передан – ответ
   `400 {"detail": "Idempotency-Key header is required"}`, сага не запускается.
2. Валидировать тело запроса (обязательные поля, `amount` – целое `> 0`);
   при нарушении – `422` (стандартный ответ валидации FastAPI).
3. Найти ключ в таблице `idempotency_keys`. Если ключ уже известен – вернуть
   `202` с ранее созданным `payment_id` и `status=duplicate`; новый платёж не
   создаётся, событие не публикуется (идемпотентность входа, см. ADR-0003).
4. Ключ новый – сгенерировать `payment_id` (UUID v4) и в **одной транзакции БД**:
   - вставить `idempotency_keys (key → payment_id)` через
     `ON CONFLICT (key) DO NOTHING` – защита от гонки параллельных запросов с
     одинаковым ключом; если вставку выиграл конкурирующий запрос, вернуть его
     `payment_id` со `status=duplicate`;
   - создать стартовую запись саги `saga_state (status=INITIATED)`.
5. Опубликовать событие `payment.initiated` в Kafka: конверт `EventEnvelope`,
   полезная нагрузка `TransferData`, ключ сообщения – `from_account`
   (события одного счёта-источника обрабатываются строго по порядку, ADR-0002).
6. Ответить `202 {payment_id, status=accepted}`. Дальнейшая обработка –
   асинхронная сага: антифрод → леджер → уведомление
   (см. [sequence-диаграммы](../sequence/happy_path.puml)); текущее состояние
   доступно методом [GET /transfers/{payment_id}](rest-get-transfer-status.md).
