## Контекст

FinTrack v2 – событийная система P2P-переводов. Клиент взаимодействует с ней
через REST API сервиса `gateway`; сам перевод выполняется асинхронно –
распределённой сагой с оркестрацией (антифрод-проверка → двойная запись в
леджере → уведомление). Ответ `202 Accepted` означает «заявка принята», итог
(`COMPLETED`/`FAILED`) появляется в статусе перевода по завершении саги.

Деньги во всех контрактах – целые числа в минорных единицах валюты (копейки):
1500,00 ₽ = `150000`. Пользовательского интерфейса в системе нет (взаимодействие
через API), поэтому разделы «Атрибуты полей форм» и «Прототипы пользовательских
интерфейсов» в документ не включены; атрибуты запроса описаны в спецификациях
REST API (см. «Связанные документы»).

## Описание пользовательской истории

| | |
|---|---|
| **Я как** | клиент (владелец счёта) |
| **Хочу** | перевести деньги со своего счёта на счёт другого клиента |
| **Для того, чтобы** | быстро рассчитаться с получателем, не зная банковских реквизитов и не посещая отделение |

## Критерии приёмки

| № | Если `<Предусловие>` | Когда `<Действие>` | Тогда `<Результат>` |
|---|----------------------|--------------------|---------------------|
| **Успешный перевод** | | | |
| 1.1 | счета отправителя и получателя существуют **и** баланс счёта-источника ≥ суммы перевода **и** перевод не нарушает правил антифрода | клиент отправляет `POST /transfers` с новым `Idempotency-Key` и валидным телом (`from_account`, `to_account`, `amount > 0`) | система отвечает `202 {payment_id, status=accepted}` **и** сага достигает статуса `COMPLETED` **и** баланс источника уменьшается на сумму перевода, баланс получателя увеличивается на ту же сумму **и** в журнале появляются ровно две проводки (`debit` источника + `credit` получателя) **и** записывается уведомление со статусом `completed` |
| 1.2 | перевод завершён | клиент запрашивает `GET /transfers/{payment_id}` | система возвращает `200` со `status=COMPLETED` и `reason=null` |
| **Повторная отправка запроса (идемпотентность)** | | | |
| 2.1 | перевод с ключом `K` уже создан | клиент повторно отправляет `POST /transfers` с тем же `Idempotency-Key: K` (ретрай по таймауту) | система возвращает `202` с **тем же** `payment_id` и `status=duplicate` **и** второй перевод не создаётся **и** событие `payment.initiated` повторно не публикуется **и** деньги не списываются дважды |
| **Отклонение антифродом: лимит суммы** | | | |
| 3.1 | сумма перевода превышает лимит одной операции (5 000,00 ₽ = `500000` копеек) | сага доходит до антифрод-проверки | сага переходит в `FAILED` с `reason=amount_limit` **и** денежные проводки не создаются **и** записывается уведомление об отказе |
| **Отклонение антифродом: велосити** | | | |
| 3.2 | с одного счёта-источника за последние 60 секунд уже выполнено 3 попытки перевода | клиент инициирует следующую попытку | сага переходит в `FAILED` с `reason=velocity` **и** денежные проводки не создаются **и** записывается уведомление об отказе |
| **Недостаточно средств** | | | |
| 4.1 | баланс счёта-источника меньше суммы перевода | сага доходит до проводки в леджере | сага переходит в `FAILED` с `reason=insufficient_funds` **и** проводки не создаются (частичная двойная запись невозможна) **и** записывается уведомление об отказе |
| **Невалидный запрос** | | | |
| 5.1 | – | клиент отправляет `POST /transfers` без заголовка `Idempotency-Key` | система отвечает `400` **и** сага не запускается |
| 5.2 | – | клиент отправляет `POST /transfers` с `amount ≤ 0` или без обязательного поля тела | система отвечает `422` **и** сага не запускается |
