# Функциональные требования (FR) – FinTrack v2

## 1. Описание системы

FinTrack v2 – событийная система **P2P-переводов** между счетами. Единственная
синхронная операция – приём заявки по REST (`gateway`); дальше перевод идёт
распределённой транзакцией-**сагой** с оркестрацией: проверка антифрода →
двойная запись в леджере → уведомление об исходе. Сервисы общаются только
через Apache Kafka. Все суммы – целые минорные единицы (копейки). Результат
перевода – терминальный статус саги (`COMPLETED` / `FAILED`) с причиной отказа.

Границы системы (scope):
- **В scope**: создание перевода, идемпотентность, антифрод-проверка, двойная
  запись и баланс, уведомление об исходе, просмотр статуса и балансов.
- **Вне scope**: аутентификация/KYC, реальный процессинг карт, мультивалютная
  конвертация, начисление комиссий, реверс/чарджбэк.

## 2. Акторы

| Актор | Тип | Интерес |
|---|---|---|
| Клиент (плательщик) | внешний | Создать перевод, узнать статус и баланс |
| Получатель | внешний | Получить деньги и уведомление |
| Антифрод-правила | внутренний сервис | Разрешить/отклонить перевод по риску |
| Леджер | внутренний сервис | Корректно провести двойную запись |
| Оркестратор саги | внутренний сервис | Провести перевод по шагам, откатить при отказе |
| Аналитика / Аудит | внешний подписчик | Потреблять терминальные события |

## 3. User stories

- **US-1.** Как клиент, я хочу перевести деньги другому счёту, указав счёт-источник,
  счёт-получатель и сумму, чтобы отправить платёж.
- **US-2.** Как клиент, я хочу, чтобы повторная отправка того же запроса
  (по `Idempotency-Key`) не создавала второй перевод, чтобы не списать деньги дважды.
- **US-3.** Как клиент, я хочу видеть статус перевода (состояние саги), чтобы
  понимать, прошёл он, отклонён или в обработке.
- **US-4.** Как клиент, я хочу видеть баланс своих счетов, чтобы контролировать
  средства.
- **US-5.** Как получатель, я хочу получать уведомление об исходе перевода, чтобы
  знать о зачислении или отказе.
- **US-6.** Как комплаенс/антифрод, я хочу отклонять подозрительные переводы по
  правилам (велосити, лимиты), чтобы снижать риск мошенничества.
- **US-7.** Как бухгалтерия/аудит, я хочу, чтобы каждый перевод порождал
  сбалансированные проводки (debit = credit), чтобы баланс всегда сходился.
- **US-8.** Как аналитик, я хочу потреблять терминальные события `payment.completed/failed`,
  чтобы строить отчётность, не затрагивая основной поток.

> Ключевая история US-1 расписана детально (Если/Когда/Тогда, по шаблону
> user story) в [`user-story-transfer.md`](user-story-transfer.md).

## 4. Use cases

### UC-1. Перевод денег (основной use case)

- **Актор:** Клиент
- **Предусловия:** счета `from_account` и `to_account` существуют; у клиента есть
  `Idempotency-Key`.
- **Основной поток (happy path):**
  1. Клиент шлёт `POST /transfers` с `Idempotency-Key` и телом
     `{from_account, to_account, amount, currency}`.
  2. Gateway валидирует тело (`amount` – целое > 0), резолвит идемпотентность:
     ключ новый → создаёт `payment_id`, пишет `idempotency_keys` и стартовую
     запись `saga_state (INITIATED)`.
  3. Gateway публикует `payment.initiated` (key=`from_account`), отвечает
     `202 {payment_id, status=accepted}`.
  4. Orchestrator создаёт `saga_state=ANTIFRAUD_PENDING`, шлёт
     `antifraud.check.requested`.
  5. Antifraud одобряет → `antifraud.check.completed {approved=true}`.
  6. Orchestrator → `saga_state=LEDGER_PENDING`, шлёт `ledger.transfer.requested`.
  7. Ledger в одной транзакции пишет `debit(from)` + `credit(to)` + строку outbox;
     relay публикует `ledger.transfer.completed {success=true}`.
  8. Orchestrator → `saga_state=COMPLETED`, публикует `payment.completed` и
     `notification.requested {status=completed}`.
  9. Notification пишет журнал и доставляет уведомление получателю.
- **Постусловия:** баланс источника уменьшен, получателя увеличен на `amount`;
  статус `COMPLETED`; отправлено уведомление.

#### Альтернативные / исключительные потоки

- **AF-1. Дубликат по `Idempotency-Key`** (см. UC-2): на шаге 2 ключ уже есть →
  возвращается тот же `payment_id` со `status=duplicate`, новый перевод не
  создаётся, событие повторно не публикуется.
- **AF-2. Отказ антифрода:** на шаге 5 `approved=false` → orchestrator ставит
  `saga_state=FAILED (reason)`, публикует `payment.failed` и
  `notification.requested {status=failed}`. Леджер не вызывается, проводок нет.
- **AF-3. Недостаточно средств в леджере:** на шаге 7 баланс источника < `amount`
  → проводки не создаются, `ledger.transfer.failed {reason="insufficient_funds"}`
  → orchestrator ставит `FAILED`, уведомляет об отказе. Компенсация не нужна
  (денежных изменений не было).
- **AF-4. Невалидный запрос:** `amount <= 0` или нет обязательного поля/заголовка
  → gateway отвечает `400`/`422`, сага не запускается.
- **AF-5. Сбой обработки события:** консьюмер ретраит до `maxRetries=3`; при
  устойчивом отказе сообщение уходит в `<topic>.dlq`, партиция не блокируется.
- **AF-6. Повторная доставка события (at-least-once):** обработчики идемпотентны;
  в леджере дубль гасит `UNIQUE(payment_id, account_id, direction)`.

### UC-2. Повторная отправка с тем же ключом

- **Актор:** Клиент (ретрай по таймауту)
- **Поток:** `POST /transfers` с уже использованным `Idempotency-Key` → gateway
  находит `payment_id` в `idempotency_keys` → возвращает тот же `payment_id`,
  ничего не публикует.
- **Постусловие:** ровно один перевод на ключ.

### UC-3. Просмотр статуса перевода

- **Актор:** Клиент. `GET /transfers/{payment_id}` → статус и шаг саги, параметры
  перевода и причина отказа (из таблицы `saga_state`). Если не найден → `404`.

### UC-4. Просмотр балансов

- **Актор:** Клиент. `GET /accounts` → список счетов и балансов из вью `balances`
  (в минорных единицах).

## 5. Правила и приоритизация (MoSCoW)

### Must have
- **FR-M1.** Создание перевода через `POST /transfers` с обязательным `Idempotency-Key`.
- **FR-M2.** Идемпотентность входа: один `Idempotency-Key` → один `payment_id`.
- **FR-M3.** Идемпотентность проводок: `UNIQUE(payment_id, account_id, direction)`.
- **FR-M4.** Двойная запись: каждый перевод = `debit(from)` + `credit(to)` на
  равную сумму, в одной транзакции БД.
- **FR-M5.** Суммы – целые минорные единицы, `amount > 0`.
- **FR-M6.** Антифрод-проверка перед проводкой; при `approved=false` – перевод
  `FAILED`, леджер не вызывается.
- **FR-M7.** Отказ леджера при недостатке средств → перевод `FAILED` без проводок.
- **FR-M8.** Состояние саги хранится и доступно через `GET /transfers/{id}`.
- **FR-M9.** Порядок событий по счёту: ключ партиционирования = `from_account`.
- **FR-M10.** Outbox в леджере: проводки и событие – атомарно.
- **FR-M11.** Уведомление об исходе перевода (`completed`/`failed`).

### Should have
- **FR-S1.** DLQ `<topic>.dlq` после `maxRetries=3`.
- **FR-S2.** `GET /accounts` с балансами.
- **FR-S3.** Терминальные события `payment.completed/failed` для внешних подписчиков.
- **FR-S4.** Сквозная трассировка через `trace_id` в конверте события.

### Could have
- **FR-C1.** Очистка/архивация старых `idempotency_keys` и опубликованных строк outbox.
- **FR-C2.** Конфигурируемые правила антифрода (лимиты, велосити).
- **FR-C3.** Просмотр истории проводок по счёту.

### Won't have (в текущей версии)
- **FR-W1.** Аутентификация/KYC, авторизация по ролям.
- **FR-W2.** Мультивалютная конвертация и комиссии.
- **FR-W3.** Реверс/чарджбэк перевода.
