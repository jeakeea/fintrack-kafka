# ADR-0003. Стратегия идемпотентности (двухуровневая)

Статус: принято · Дата: июль 2026 · Область: gateway, ledger, все консьюмеры

## Контекст

Доставка в Kafka – **at-least-once**: оффсет коммитится только после успешной
обработки (`shared/kafka_client.py`), поэтому сообщение может быть доставлено
повторно (ретрай, ребаланс, перезапуск). Клиент тоже может ретраить HTTP-запрос
по таймауту. Для денег повторная обработка не должна приводить к двойному
списанию или дублю платежа.

## Решение

Идемпотентность на **двух уровнях**, на естественных границах:

1. **Вход (gateway), бизнес-уровень** – по заголовку `Idempotency-Key`.
   Таблица `idempotency_keys (key PRIMARY KEY, payment_id)`: первый запрос с
   ключом создаёт `payment_id` и публикует `payment.initiated`; повторный запрос
   с тем же ключом возвращает **тот же** `payment_id` и не запускает сагу заново.
2. **Леджер, уровень данных** – по `UNIQUE (payment_id, account_id, direction)`
   в `ledger_entries`. Даже если событие `ledger.transfer.requested` придёт
   дважды, повторная вставка debit/credit нарушит уникальность и будет
   проигнорирована – проводки не задвоятся.

Все консьюмеры обязаны быть **идемпотентными** (безопасны к повторной обработке),
т.к. транспорт at-least-once.

## Альтернативы

- **Exactly-once (Kafka transactions / EOS)**: дороже, сложнее, и всё равно не
  покрывает идемпотентность относительно внешнего клиента и БД. Двухуровневые
  естественные ключи проще и надёжнее для денег.
- **Дедуп только на входе**: не защищает от дублей внутри саги (ретраи
  консьюмера леджера). Недостаточно.
- **Дедуп по `event_id` в общей таблице обработанных сообщений**: рабочий
  паттерн, но добавляет запись на каждое событие; для леджера естественный
  доменный ключ `(payment_id, account_id, direction)` точнее и бесплатен.

## Последствия

- (+) Безопасные ретраи на всех границах: HTTP, шина, БД.
- (+) Никаких двойных списаний и дублирующих платежей.
- (−) Клиент обязан присылать корректный `Idempotency-Key` (требование API).
- (−) `idempotency_keys` растёт – нужна политика TTL/архивации старых ключей.
- (−) Обработчики должны проектироваться идемпотентными осознанно (код-ревью,
  тесты на повторную доставку).
