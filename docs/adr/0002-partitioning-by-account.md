# ADR-0002. Партиционирование по счёту-источнику (ordering)

Статус: принято · Дата: июль 2026 · Область: Kafka, все продюсеры

## Контекст

Деньги требуют предсказуемого порядка операций по одному счёту: два перевода
с одного счёта не должны «обгонять» друг друга и приводить к гонкам при расчёте
баланса. Kafka гарантирует порядок **только в пределах одной партиции**, а
распределение по партициям определяется ключом сообщения.

## Решение

Ключ партиционирования для всех событий денежного потока – **`from_account`**
(счёт-источник). Все события одного счёта-источника попадают в одну партицию и
обрабатываются строго последовательно.

`shared/topics.py` фиксирует это в контракте; продюсеры (`gateway`,
`orchestrator`, `ledger`) передают `key=from_account` в `Producer.send(...)`.

## Альтернативы

- **Ключ = `payment_id`**: гарантирует порядок только в рамках одного платежа,
  но разные платежи одного счёта могут обрабатываться параллельно → гонки на
  балансе и нарушение причинно-следственного порядка по счёту. Отвергнуто.
- **Без ключа (round-robin)**: максимальный параллелизм, но никакого порядка –
  недопустимо для денег.
- **Ключ = `to_account`**: упорядочивает по получателю; источник списания может
  «обгонять» сам себя. Для модели «списание = критичная операция» источник важнее.

## Последствия

- (+) Строгий per-account порядок без распределённых блокировок.
- (+) Балансовые инварианты считаются на последовательном потоке проводок счёта.
- (−) Параллелизм ограничен числом партиций и распределением счетов по ним;
  «горячий» счёт (много переводов) упирается в одну партицию (hot partition).
- (−) Число партиций топика задаёт верхний предел параллелизма консьюмер-группы;
  изменение количества партиций меняет раскладку ключей – планируется заранее.
- (−) Порядок гарантирован **в пределах партиции**, а не глобально по всей
  системе – это сознательный размен в пользу масштабируемости.
