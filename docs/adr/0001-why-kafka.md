# ADR-0001. Шина событий: Apache Kafka

Статус: принято · Дата: июль 2026 · Область: вся система

## Контекст

P2P-перевод проходит несколько асинхронных шагов (gateway → orchestrator →
antifraud → ledger → notification). Нужен транспорт между сервисами, который
обеспечивает: гарантированную доставку, **строгий порядок по счёту**,
переобработку при сбоях, развязку производителей и потребителей, наблюдаемость
потока событий и возможность подключать новых подписчиков (аналитика, аудит)
без изменения отправителя.

## Решение

Использовать **Apache Kafka** (KRaft-режим, без ZooKeeper) как шину событий.
Каждый шаг саги – отдельный топик (9 топиков, см. `shared/topics.py`).

Почему Kafka подходит под задачу:

- **Партиционирование и порядок**: ключ = `from_account` → события одного счёта
  идут в одну партицию и обрабатываются строго по порядку (см. ADR-0002).
- **Durable log + смещения**: at-least-once, ручной commit оффсета только
  после успешной обработки (`enable_auto_commit=False`), переобработка с
  `auto_offset_reset=earliest`.
- **Consumer groups**: горизонтальное масштабирование консьюмеров без потери
  порядка в пределах партиции.
- **Fan-out без связности**: терминальные `payment.completed/failed` читают
  внешние подписчики, не затрагивая отправителя.
- **Retention**: лог событий = аудит-трейл и основа для replay.

## Альтернативы

- **RabbitMQ (AMQP)**: отличная маршрутизация и per-message ack, но порядок
  при нескольких консьюмерах и конкурентной доставке не гарантируется так же
  естественно, как партиции Kafka; нет встроенного durable replay лога. Для
  «ordering по счёту» Kafka выигрывает.
- **Синхронный REST между сервисами**: простейшая модель, но жёсткая связность
  и temporal coupling – если antifraud/ledger недоступен, падает весь перевод;
  нет буферизации пиков, ретраи и идемпотентность ложатся на каждого вызывающего.
- **Postgres как очередь (`SELECT … FOR UPDATE SKIP LOCKED`)**: годится для
  outbox внутри сервиса, но не масштабируется как межсервисная шина и не даёт
  партиционирования по ключу из коробки.

## Последствия

- (+) Развязка сервисов, устойчивость к пикам и временной недоступности.
- (+) Гарантия порядка по счёту, durable-аудит, простое подключение подписчиков.
- (−) At-least-once требует **идемпотентных обработчиков** (см. ADR-0003) и DLQ
  (`<topic>.dlq` после `maxRetries=3`).
- (−) Eventual consistency: статус перевода становится терминальным асинхронно
  (см. NFR по консистентности).
- (−) Операционная сложность: брокер, мониторинг lag, управление топиками.
